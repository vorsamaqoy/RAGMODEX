# MolChat — Design Context

## Design Context

### Users
Mixed audience spanning academic researchers, pharmaceutical/industry medicinal chemists, and cheminformatics students. Experts need density and speed; students need legibility and discoverability. Neither group tolerates decoration — both groups tolerate complexity when it is well-organized. Context of use: focused desktop sessions, often alongside a lab notebook, model training script, or paper. Not mobile-first.

### Brand Personality
**Useful, fast, intuitive.**
The interface should feel like a well-engineered research instrument — authored, not assembled. It builds trust through clarity and consistency, not through visual flair. Think: the confidence of a well-designed scientific software (alvaDesc, KNIME) rather than the polish of a SaaS dashboard. Data is the hero; the UI is the frame.

### Aesthetic Direction
- **Reference:** alvaDesc + KNIME — structured, information-dense, professionally composed. Clean panel hierarchy, confident typographic scale, muted palette with purposeful accent use.
- **Anti-reference:** Generic AI dashboards (dark mode + neon glows + gradient cards), academic-lab aesthetics (gray Bootstrap tables, no visual hierarchy), SaaS marketing sites (hero metrics, gradient text, glassmorphism).
- **Theme:** Dark, always. The jade-dark + champagne gold palette is established and should be respected. The gold accent earns its place on interactive elements and key data points — not decoration.
- **Tone:** Dense but legible. Confident but not loud. Every visual element should feel like it was placed deliberately.

### Design Principles
1. **Data first** — Molecular structure images, P(active) scores, and SHAP values are the primary content. UI chrome serves them; it never competes.
2. **Progressive disclosure** — Default state shows what's needed to act. Advanced settings collapse behind one interaction. Students see a clear path; experts find depth quickly.
3. **Consistent token use** — All colors from CSS custom properties. No hard-coded hex in component code. This is non-negotiable for a tool with this many data states (active/inactive/ambiguous/out-of-domain).
4. **Legible at density** — Font sizes never below 0.7rem for data labels. SMILES strings render in the app font (Space Grotesk), not monospace. Scientific notation is always human-readable.
5. **Trust through restraint** — No decorative gradients, no glow effects, no motion for its own sake. Interactions confirm state; they don't entertain.
